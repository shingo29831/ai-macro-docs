7. ディレクトリ構造と開発要件 (ADR)

本システムのUIとロジックを完全に疎結合化し、各認識エンジンや自己修復プロセスを関心事ごとに分離するための全体アーキテクチャおよび開発ルールを定義します。

7.1. ディレクトリ構成

ai-macro-system/
├── src/
│   ├── main.py                  // アプリケーションのエントリポイント
│   ├── ui/                      // 1画面構造・ウィジェットを管理するUI層
│   │   ├── views/               // MainWindow, RecordDialogなどのビュークラス
│   │   ├── viewmodels/          // UIの状態とビジネスロジックの橋渡し
│   │   └── resources/           // .uiファイルや.cssファイルを格納
│   │       ├── ui/
│   │       └── css/
│   ├── core/                    // ビジネスロジック層
│   │   ├── recorder/            // 操作記録（OS Hook）および入力操作データの取得
│   │   ├── generator/           // 統合データからのマクロコードAI生成
│   │   ├── executor/            // 動的待機・即時ポーリングによる実行エンジン
│   │   └── healer/              // 4段階自己修復（Image Diff検証・動的再生成）エンジン
│   ├── engines/                 // 外部認識エンジン層
│   │   ├── yolo/                // UI要素認識エンジン
│   │   └── ocr/                 // テキスト認識エンジン
│   ├── models/                  // workflow_IDや統合データのデータ構造定義
│   └── utils/                   // 環境変数管理、ロガーなどの共通ユーティリティ
├── tests/                       // 振る舞い・エラーテスト群
│   ├── ui/                      // UI状態遷移の検証
│   ├── core/                    // ロジックと例外処理の検証
│   └── integration/             // 自己修復フローのエンドツーエンド検証
├── macros/                      // 生成されたマクロごとにリソースをカプセル化するディレクトリ
│   └── wf_template/             // workflow.jsonやimagesを格納
├── docs/                        // HTML仕様書群およびADR
├── .env.example                 // 環境変数のテンプレート
├── requirements.txt             // 依存パッケージ
└── README.md                    // 実例込みのドキュメント

7.2. アプリケーション開発要件

【基本・設計】

  • ファイル責務の明示: 全てのファイルの冒頭にAI向け役割コメント（例: @role: ...）を記述し、関心事ごとにファイルを細かく分割する。
  • 疎結合の徹底: PySide6を用いたUI層と、生成・実行などのビジネスロジック層を完全に分離する。
  • コードの意図: 処理の「何」はコードの命名や構造で表現し、コメントには「なぜその実装にしたのか（背景）」のみを記述する。

【品質・保守】

  • 命名とDRY原則: DRY原則を徹底し、変数や関数の命名から推測可能な明確な構造を維持する。
  • ボーイスカウトルール: 機能の追加・修正時には、周辺に存在する既存の技術的負債も同時に修正する。
  • 非推奨の禁止: 非推奨（Deprecated）なライブラリや関数の使用は禁止。必須時は代替案と理由を提示し要承諾。

【セキュリティ・堅牢性】

  • 機密情報の保護: APIキー等の環境依存情報は .env を用いて環境変数化し、ハードコードを厳禁とする。
  • 入力のサニタイズ: 各認識エンジン（YOLO、OCR）やOS Hookからの生データは、必ず検証・サニタイズを行ってから統合データとして処理する。
  • エラーハンドリング: エラー握り潰しは厳禁。画像差分超過や対象UI見失い時は、4段階自己修復ツリーに従い自律対応、または安全に強制終了させる。
  • 外部通信の失敗前提: AIコアへのリクエスト等、外部通信は失敗を前提としタイムアウトとリトライ機構を必ず実装する。

【テスト・運用】

  • 振る舞いのテスト: 内部実装詳細（プライベートメソッド等）には依存せず、「マクロの実行という振る舞い」と「異常発生時のエラー検知・修復・安全停止」が正しく機能するかを tests/ 以下の各層（ui, core, integration）で検証する。
  • ドキュメント運用: 実例を含めたREADMEを作成し、重要設計はADR（Architecture Decision Records）として提案・蓄積する。
